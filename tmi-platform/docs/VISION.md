# TMI Vision Blueprint (Future-State)

This document captures aspirational architecture and direction.  
It is intentionally separated from `README.md` to avoid mixing future-state design with current operational reality.

---

## Platform intent

A persistent creator-first arena combining:

- live performance surfaces
- audience participation systems
- automation/bot orchestration
- economy and payout engines
- operator-grade monitoring and guardrails

---

## High-level architecture vision

```text
Entry/Auth → Arena/Lobby Surfaces → Live Rooms/Challenges → Economy Loop → Payout/Operations
```

### Candidate subsystems in long-form vision

- Guided freedom navigation layer
- Unified micro-transaction economy spine
- Split settlement/revenue dispatch engines
- Live audience realism + performer confidence systems
- Operator/overseer command surfaces
- Bot triage + assisted support pipelines
- Seasonal issue/rotation-driven content/economy loops

---

## Financial blueprint (vision-layer)

Targeted settlement style (conceptual):

1. Gross transaction
2. Processor fees
3. Reserve layer
4. Operating pool
5. Distribution rules (creator/platform/admin/growth/future funds)

Implementation for production must be compliance-first and processor-supported.

---

## Guardrail principles

- No dead-end navigation
- No orphan routes
- Runtime friction telemetry for broken movement
- Progressive fallback behavior for missing dependencies

---

## Experience principles

- Movement should feel alive and continuous
- Lobbies should signal activity and intent clearly
- Upgrades/pricing should create immediate perceived value
- Core actions must be obvious in first-session UX

---

## Governance rule for docs

- `README.md` = reality now
- `docs/ONBOARDING.md` = operator usage now
- `docs/VISION.md` = target architecture and long-horizon system design

This separation prevents operator confusion during active soft-launch phases.
