# REAL VENUE SYSTEM
## BerntoutGlobal XXL — TMI Platform
## Effective: 2026-03-23 | Status: LOCKED

---

## PURPOSE

Bind digital rooms to real venue identity and operations so rooms are world-based, not abstract containers.

---

## VENUE CORE MODEL (TARGET)

A venue record should define:
- venue name
- venue type (club, studio, arena, lounge, etc.)
- city/region
- capacity tiers
- operator/manager
- sponsor inventory
- visual template binding
- event schedule linkage

---

## DIGITAL TWIN REQUIREMENT

Every major room may optionally attach to a venue twin:
- venue identity
- stage style/theme
- sponsor board defaults
- room composition template
- moderation profile

---

## PROVISIONING CHAIN

Venue/event workflow:
1. Venue selected
2. Event created
3. Room instance provisioned
4. Room template applied
5. A/V profile applied
6. Host/mod roles assigned
7. Sponsor slots assigned
8. Archive target assigned

---

## OPERATIONS INTEGRATION

Room runtime must be able to read venue-linked configuration for:
- queue policy
- media policy
- moderation strictness
- recording/replay mode
- access/ticket mode

---

## ADMIN REQUIREMENTS

Venue operator controls must include:
- room open/close
- schedule update
- host/mod assignment
- sponsor slot management
- emergency lock/freeze

---

## STATUS RULE

A room is not "venue wired" unless it has:
- venue identity binding
- provisioning path
- operator control path
- archive/history linkage
